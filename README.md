# Residual-CNN

**A CNN-Residual deep neural network for blind channel coefficient (h_k) estimation in DVB-S2 satellite links.**

Residual-CNN takes the pilot symbols extracted from a DVB-S2 physical-layer frame (plus link/MODCOD side-information) and predicts the complex channel coefficient `h_k = h_real + j¬∑h_imag` ‚Äî the quantity normally estimated via pilot-aided least-squares/ML methods, but learned here end-to-end from data generated by `dataset_gen.m`.

The name reflects the architecture's core building block: a stack of **dilated 1D convolutional residual blocks** that wind through the pilot sequence like a helix, progressively widening their receptive field (dilations 1 ‚Üí 2 ‚Üí 4 ‚Üí 8) to capture both local pilot-symbol patterns and the global structure of the pilot block across the frame.

## Problem Background

`dataset_gen.m` simulates a DVB-S2 forward link through an ITU-R P.618 rain-fading + AWGN channel for 8 different MODCODs (e.g. `QPSK_1_3`, `32APSK_5_6`, ...). For each transmitted frame, it:

1. Generates a DVB-S2 physical layer frame for the chosen MODCOD.
2. Passes it through a rain-faded, noisy satellite channel with a known ground-truth complex gain `H_true = h_real + j¬∑h_imag`.
3. Receives and matched-filters the waveform, extracts the **pilot symbols** embedded in the frame (DVB-S2 pilot blocks), and normalizes them.
4. Writes one CSV row per simulated frame containing: simulation metadata (MODCOD, Es/No, rain attenuation, noise variance, etc.), the **ground-truth channel coefficient** (`htrue_real`, `htrue_imag`), and the **received pilot symbols** (`pilot_re_i`, `pilot_im_i`, `pilot_mask_i` for `i = 1..Np`).

**Goal:** train a neural network that maps the received pilot symbols + side-info ‚Üí an estimate of `(htrue_real, htrue_imag)`, learning an implicit channel estimator directly from simulated data.

## Dataset Format

Each MODCOD gets its own CSV file (8 files total, one per MODCOD), with this column layout:

```
[ 25 metadata columns ] + pilot_re_1..Np + pilot_im_1..Np + pilot_mask_1..Np
```

### Metadata columns (in order)

| # | Column | Description |
|---|---|---|
| 1 | `sample_id` | Row index within the file |
| 2 | `modcod` | MODCOD ID (1‚Äì28, DVB-S2 numbering) |
| 3 | `modcod_label` | Human-readable label, e.g. `QPSK_1_3`, `32APSK_5_6` |
| 4 | `channel_type` | ITU-R climate zone (e.g. `tropical`) |
| 5 | `modOrder` | Modulation order (4, 8, 16, 32) |
| 6 | `codeRate_num` | Code rate numerator |
| 7 | `codeRate_den` | Code rate denominator |
| 8 | `plFrameSize` | Total PL frame length (symbols) |
| 9 | `Np_actual` | Number of pilot symbols for this MODCOD |
| 10 | `numPilotBlks` | Number of pilot blocks in the frame |
| 11 | `esno_nom_dB` | Nominal Es/No sweep value (dB) |
| 12 | `rainAtt_dB` | Sampled rain attenuation (dB) |
| 13 | `p_exceedance` | Rain exceedance probability used |
| 14 | `snr_eff_dB` | Effective Es/No after rain (dB) |
| 15 | `snr_awgn_dB` | AWGN SNR applied at the channel (dB) |
| 16 | `nvar_pilot` | Pilot-based noise variance estimate |
| 17 | `pilotScale` | Normalization scale applied to received pilots |
| 18 | `Ap_linear` | Linear rain attenuation factor |
| 19 | `phi_rad` | Channel phase (rad) |
| 20 | `b_gain` | Antenna/beam gain factor |
| 21 | `b_max` | Link budget constant |
| 22 | **`htrue_real`** | **Ground-truth channel coefficient ‚Äî real part (TARGET)** |
| 23 | **`htrue_imag`** | **Ground-truth channel coefficient ‚Äî imag part (TARGET)** |
| 24 | `htrue_mag` | `\|H_true\|` (derived, not used as a feature) |
| 25 | `htrue_phase_rad` | `angle(H_true)` (derived, not used as a feature) |

### Pilot columns (repeated for `i = 1..Np`)

| Column | Description |
|---|---|
| `pilot_re_i` | Real part of the i-th received, normalized pilot symbol |
| `pilot_im_i` | Imaginary part of the i-th received, normalized pilot symbol |
| `pilot_mask_i` | 1 if this pilot slot is real data, 0 if it's padding |

### Example: the two provided sample files

| File | MODCOD | `Np_actual` | Total columns |
|---|---|---|---|
| `QPSK_1_3.csv` | QPSK, rate 1/3 | 792 | 2401 = 25 + 3√ó792 |
| `32APSK_5_6.csv` | 32APSK, rate 5/6 | 288 | 889 = 25 + 3√ó288 |

This illustrates the core challenge tackled by `preprocess.py`: **every MODCOD has a different `Np`**, so the 8 CSVs have different numbers of columns and cannot simply be concatenated.

## The Variable-Pilot-Length Problem

DVB-S2 pilot block count scales with frame size and MODCOD, so `Np` ranges from a few hundred to nearly a thousand symbols depending on MODCOD. A single fixed-size CNN input therefore can't directly consume all 8 files.

**Residual-CNN's solution** (implemented in `preprocess.py` / `predict.py`):

1. **Detect `Np` per file** from its column count: `Np = (n_cols - 25) / 3`.
2. **Compute `MAX_NP`** = the largest `Np` across all 8 files (this becomes the model's fixed pilot-sequence input length).
3. **Zero-pad** `pilot_re`, `pilot_im`, and `pilot_mask` for every shorter-`Np` file up to `MAX_NP`. The `pilot_mask` channel is `0` for all padded positions and `1` for real pilot data.
4. **Feed the mask as a 3rd input channel** to the CNN (alongside re/im), so the network can learn to disregard padded positions ‚Äî effectively making the convolutions "MODCOD-length aware" without needing separate models per MODCOD.
5. **One-hot encode `modcod_label`** and append it (plus `Np_actual`, `numPilotBlks`, etc.) to the side-info feature vector, so the model also has explicit knowledge of which MODCOD (and therefore which real pilot length) it's looking at.

At prediction time, if a new CSV has `Np <= MAX_NP` it's padded the same way; if (unusually) `Np > MAX_NP` it's truncated to `MAX_NP`.

## Architecture: How Residual-CNN Works

Residual-CNN is a **two-branch fusion network**:

![Residual-CNN architecture overview](./docs/architecture_overview.svg)

### 1. Pilot branch

- **Input:** `[batch, MAX_NP, 3]` ‚Äî channels are `(pilot_re, pilot_im, pilot_mask)`.
- **Stem:** `Conv1D(64, kernel=7)` + BatchNorm + ReLU ‚Äî wide initial receptive field over the pilot sequence.
- **4 residual blocks**, each:
  - Two `Conv1D` layers (kernel=3) with BatchNorm + ReLU, L2-regularized.
  - **Dilation rates 1, 2, 4, 8** across the 4 blocks ‚Äî each block sees a progressively wider window of the pilot sequence without increasing parameter count, letting the network correlate pilot symbols that are far apart within a pilot block or across blocks.
  - **Skip connection** (identity, or a 1√ó1 conv projection if channel dims change) ‚Äî improves gradient flow and lets the network fall back to near-identity mappings if deeper features aren't useful.
  - Filter progression: 64 ‚Üí 64 ‚Üí 128 ‚Üí 128.

  The internal structure of each residual block (`resblock{i}_conv1` ‚Üí `resblock{i}_bn1` ‚Üí `resblock{i}_relu1` ‚Üí `resblock{i}_conv2` ‚Üí `resblock{i}_bn2` ‚Üí `Add` with the skip path ‚Üí `resblock{i}_relu2`) is shown below:

  ![Residual block internal structure](./docs/resblock_detail.svg)

- **Global pooling:** both `GlobalAveragePooling1D` and `GlobalMaxPooling1D` are computed and concatenated ‚Äî average pooling captures the overall pilot statistics (useful for noise-variance-like cues), max pooling captures salient outlier pilot symbols.
- The mask channel ensures padded positions (for MODCODs with `Np < MAX_NP`) contribute `0` to the re/im channels, and the network can learn (via the mask channel itself) to discount those positions in the pooled representation.

### 2. Side-info branch

- **Input:** scalar features ‚Äî `modOrder`, `codeRate_num`, `codeRate_den`, `plFrameSize`, `Np_actual`, `numPilotBlks`, `esno_nom_dB`, `rainAtt_dB`, `p_exceedance`, `snr_eff_dB`, `snr_awgn_dB`, `nvar_pilot`, `pilotScale`, `Ap_linear`, `phi_rad`, `b_gain`, `b_max`, plus a **one-hot encoding of `modcod_label`**.
- Two dense layers (`64` ‚Üí BatchNorm ‚Üí `32`, ReLU) compress this into a learned embedding of the operating point (MODCOD, channel condition).

### 3. Fusion head

- Pooled pilot features (`256`-d: avg + max pooled, 128 channels each) concatenated with the side-info embedding (`32`-d).
- Two dense layers (`128`, `64`, ReLU + Dropout 0.2, L2-regularized) followed by a final linear `Dense(2)` producing `[hk_real, hk_imag]`.

### Loss & metrics

- **Loss:** MSE on standardized `(hk_real, hk_imag)`.
- **Metrics:** MAE and RMSE (also computed in standardized space during training; `predict.py` inverse-transforms back to physical units).
- **Optimizer:** Adam, default LR `1e-3`, with `ReduceLROnPlateau`.

## Usage

### Step 1 ‚Äî Preprocessing

Place all 8 per-MODCOD CSV files (plus, optionally, `ref_pilots_lookup.mat` ‚Äî which is ignored) in a single directory, then run:

```bash
python preprocess.py --data_dir /path/to/dataset_output --out_dir ./data_processed
```

**What it does:**

- Scans `--data_dir` for all `*.csv` files (ignoring `ref_pilots_lookup*`).
- For each file, detects `Np` from its column count.
- Computes `MAX_NP = max(Np across all files)`.
- Zero-pads `pilot_re` / `pilot_im` / `pilot_mask` to `MAX_NP` for every file.
- Builds a unified label set from `modcod_label` and one-hot encodes it.
- Concatenates all files into one dataset, then splits into **train / val / test** (default 70% / 15% / 15%, configurable).
- Fits `StandardScaler`s on the **training split only** for:
  - side-info features (incl. one-hot modcod),
  - pilot real/imag channels (mask channel left untouched, then re-zeroed for padded positions after scaling),
  - target `(hk_real, hk_imag)`.
- Saves everything needed for training/inference: `train.npz`, `val.npz`, `test.npz`, `meta.json`, and 4 `.pkl` scaler files.

**Useful flags:**

| Flag | Default | Description |
|---|---|---|
| `--data_dir` | *(required)* | Folder containing the 8 MODCOD CSVs |
| `--out_dir` | `./data_processed` | Where to write processed data |
| `--test_size` | `0.15` | Fraction of data held out for testing |
| `--val_size` | `0.15` | Fraction of data held out for validation |
| `--seed` | `42` | Random seed for the split |

**Example output:**

```
Found 8 CSV files:
  - QPSK_1_3.csv
  - 8PSK_3_4.csv
  - ...

Detected per-file Np values: [('QPSK_1_3.csv', 792), ('32APSK_5_6.csv', 288), ...]
MAX_NP (padding target) = 792

Modcod labels found (8): ['16APSK_...', '32APSK_5_6', '8PSK_3_4', 'QPSK_1_3', ...]

Total samples: 88
Saved preprocessed dataset to: ./data_processed
  train: 61 samples
  val:   13 samples
  test:  14 samples
  pilot tensor shape: (792, 3)
  side feature dim:   25
```

### Step 2 ‚Äî Training

```bash
python train.py \
  --data_dir ./data_processed \
  --ckpt_dir ./checkpoints \
  --plot_dir ./plots \
  --epochs 200 \
  --batch_size 256
```

**What it does:**

1. **GPU setup:** lists available GPUs and enables memory growth (avoids TF grabbing all VRAM up front). Falls back to CPU automatically if no GPU is found.
2. **Loads** `train.npz` / `val.npz` and builds `tf.data.Dataset` pipelines (shuffled + batched + prefetched for train; batched only for val).
3. **Checkpoint resume logic** (see [Step 3](#step-3--resuming-training)).
4. **Builds and compiles** Residual-CNN via `model.py` if no checkpoint exists.
5. **Trains** with the following callbacks:
   - `ModelCheckpoint` ‚Üí `latest.keras` every epoch (for resuming).
   - `ModelCheckpoint` ‚Üí `best.keras` whenever `val_loss` improves.
   - `EarlyStopping` on `val_loss` (patience configurable, restores best weights).
   - `ReduceLROnPlateau` on `val_loss` (halves LR, floor `1e-6`).
   - `PlotHistoryCallback` (custom) ‚Äî appends to `history.json` and re-renders `training_curves.png` **after every epoch**.
   - `CSVLogger` ‚Üí `training_log.csv` (appends across resumed runs).
6. Saves `final_model.keras` once training stops (early stopping or max epochs reached).

**Useful flags:**

| Flag | Default | Description |
|---|---|---|
| `--data_dir` | `./data_processed` | Preprocessed data directory |
| `--ckpt_dir` | `./checkpoints` | Where checkpoints are saved/loaded |
| `--plot_dir` | `./plots` | Where plots/history/logs are saved |
| `--epochs` | `200` | Max number of epochs |
| `--batch_size` | `256` | Batch size |
| `--lr` | `1e-3` | Initial learning rate |
| `--patience` | `15` | Early-stopping / LR-plateau patience |
| `--fresh` | off | Ignore any existing checkpoint and start a brand-new model |

---

### Step 3 ‚Äî Resuming Training

Training with Residual-CNN is **crash-safe**. If training is interrupted (Ctrl-C, OOM, machine reboot, etc.), simply rerun the **same command**:

```bash
python train.py --data_dir ./data_processed --ckpt_dir ./checkpoints --plot_dir ./plots --epochs 200
```

- `train.py` checks for `checkpoints/latest.keras`. If found, it **loads the full model (weights, optimizer state, architecture)** instead of building a new one.
- It also reads `plots/history.json` to determine how many epochs have already been completed (`initial_epoch`), so the epoch counter, learning-rate schedule, early-stopping patience counter, and plots all continue seamlessly rather than restarting from epoch 0.
- To discard the checkpoint and start completely fresh, pass `--fresh`.

---

### Step 4 ‚Äî Prediction

```bash
python predict.py \
  --csv /path/to/new_modcod.csv \
  --data_dir ./data_processed \
  --ckpt_dir ./checkpoints \
  --out predictions.csv
```

**What it does:**

1. Loads `meta.json` and all 4 scalers from `--data_dir`.
2. Loads the trained model ‚Äî preferring `best.keras`, falling back to `final_model.keras`, then `latest.keras`.
3. Reads the input CSV, detects its `Np` from column count.
4. Pads (or truncates, if `Np > MAX_NP`) pilot channels to `MAX_NP`, applies the saved pilot scalers, re-zeros padded positions.
5. Builds the side-info vector (including one-hot `modcod_label` ‚Äî **must be a label seen during training**, otherwise raises an error).
6. Runs `model.predict(...)` and **inverse-transforms** the output back to physical `hk_real` / `hk_imag` units using `target_scaler`.
7. Writes a CSV with `sample_id`, `modcod`, `modcod_label`, `hk_real_pred`, `hk_imag_pred`, and ‚Äî if ground truth is present in the input ‚Äî `hk_real_true`, `hk_imag_true` for easy comparison.

## GPU Support

`train.py` calls `tf.config.list_physical_devices("GPU")` at startup:

- If one or more GPUs are found, `set_memory_growth(True)` is enabled for each (prevents TensorFlow from pre-allocating all GPU memory), and the GPU names are printed.
- If none are found, training proceeds on CPU automatically ‚Äî no code changes needed.

No manual device placement is required; Keras/TensorFlow will place ops on the GPU automatically when available.

## Extending Residual-CNN

- **More MODCODs / larger `MAX_NP`:** just rerun `preprocess.py` with the new CSVs in `--data_dir`. `MAX_NP`, the label set, and all scalers are recomputed ‚Äî but note this means **retraining from scratch** (old checkpoints won't match the new input shapes/label space). Use `--fresh`.
- **Different architecture depth/width:** edit `cnn_filters`, `dilations`, and `dense_units` in `build_model()` in `model.py`.
- **Predict magnitude/phase instead of real/imag:** add `htrue_mag`/`htrue_phase_rad` to `TARGET_COLS` in `preprocess.py` (and adjust `model.py`'s output layer size accordingly).
- **Per-MODCOD fine-tuning:** load `best.keras`, freeze the pilot-branch layers, and fine-tune only the fusion head on a single MODCOD's data for specialization.

## Troubleshooting

| Symptom | Likely cause / fix |
|---|---|
| `ValueError: Unexpected column count ... cannot infer Np` | Input CSV doesn't match the expected `25 + 3*Np` column layout ‚Äî check it came from `dataset_gen.m` and wasn't truncated. |
| `ValueError: Unknown modcod_label(s) not seen during training` | `predict.py` was given a CSV for a MODCOD not present when `preprocess.py` built the label set ‚Äî re-run preprocessing with that MODCOD's data included, then retrain. |
| Training resumes at epoch 0 even though checkpoints exist | `plots/history.json` is missing or empty ‚Äî it's used to recover the epoch count. Don't delete `plots/` between runs if you want clean resume numbering (training itself still resumes correctly from `latest.keras` either way). |
| GPU not detected | Confirm `nvidia-smi` shows the GPU and your TensorFlow build includes GPU support (`pip install tensorflow[and-cuda]` on Linux, or check CUDA/cuDNN versions against the [TF compatibility matrix](https://www.tensorflow.org/install/source#gpu)). |
| Loss is `NaN` early in training | Try a lower `--lr` (e.g. `1e-4`) ‚Äî possible with very small batch sizes or if a feature wasn't properly standardized (check `meta.json` / scaler files were regenerated after any preprocessing changes). |
